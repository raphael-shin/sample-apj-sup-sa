## Issue 1 — UTF-8 decode error mid-stream

### What we saw

In the RTVI logs, twice:
```
error: Error processing frame: 'utf-8' codec can't decode bytes in
position 1022-1023: unexpected end of data
```

Both errors were `fatal: false` — the bot recovered and responded — but
some of the agent's text response is being silently dropped.

### Why this happens

`server/analytics_processor.py:101` does `buf += chunk.decode("utf-8")`
on each HTTP response chunk. If a multi-byte UTF-8 character (em-dash,
smart quote, currency symbol, emoji) straddles a chunk boundary, the
naive `.decode("utf-8")` raises `UnicodeDecodeError` because the chunk
ends mid-codepoint. Position 1022-1023 is exactly where AWS/aiobotocore
splits chunks.

### Fix (one of ours, not infra)

Use an `IncrementalDecoder`. It buffers partial bytes internally and
emits codepoints only when complete.

```python
# server/analytics_processor.py

# Add to imports:
import codecs

# Inside process_frame, replace:
buf = ""
async for chunk in response.get("response", []):
    buf += chunk.decode("utf-8")
    while "\n" in buf:
        ...

# With:
decoder = codecs.getincrementaldecoder("utf-8")()
buf = ""
async for chunk in response.get("response", []):
    buf += decoder.decode(chunk)
    while "\n" in buf:
        ...
# After the loop (before the existing `if buf:` block):
buf += decoder.decode(b"", final=True)
```

This is a Pipecat-side change only — no redeploy needed; restart the
local `bot.py` and the bug is gone.

### How to verify

Replay the same question that triggered the error ("Can you give me a
report of the top revenue bookings?"). The decode error should not
recur. The bot's response should also be slightly more complete since
no bytes are being silently dropped at chunk boundaries.

## Issue 2 — Bot returns canned summary instead of answering the question

### What we saw

After several successful turns, the user asked:
1. "the top revenue customers data that you just provided. Can you
   break it down by month?" (10:20:33)
2. "How about the lowest performing unicorns?" (10:23:45)

Both times the bot ignored the actual question and returned the same
generic top-level summary ("six thousand nine hundred sixty-three
bookings generating eighteen point eight million dollars..."). It even
appended "Would you like me to break that down by month, customer, or
unicorn breed?" — i.e. the bot acted as if the conversation had just
restarted.

### Hypotheses, ranked

1. **Tool call failed silently and the agent fell back to a memory
   summary.** Most likely. Strands' `stream_async` can swallow tool
   errors and the agent compensates with a high-level recap.
2. **Memory replay collision** — old `[[plain-text]]`-formatted memory
   events from before our fix #2 are being loaded alongside new
   JSON-formatted ones, confusing the agent. (Our loader has a
   plain-text fallback, but mixing formats may still confuse the
   model.)
3. **Guardrail input redaction** — the input message hit a guardrail
   topic filter and got redacted to the canned "I can only help with..."
   response, but routed through the regular response path instead of
   the redaction path. (Less likely given the bot didn't say the
   redaction phrase.)

### How to triage

Check the AgentCore Runtime CloudWatch logs for the relevant timestamps
(10:21:17 and 10:23:59 in the test transcript). Look for:

```bash
RUNTIME_ID=$(aws cloudformation describe-stacks \
  --stack-name agentic-analytics-demo --region us-west-2 \
  --query "Stacks[0].Outputs[?OutputKey=='AgentRuntimeId'].OutputValue" \
  --output text)

# AgentCore Runtime logs are under /aws/bedrock-agentcore/runtimes/<id>
aws logs tail "/aws/bedrock-agentcore/runtimes/$RUNTIME_ID-DEFAULT" \
  --region us-west-2 --since 1h --format short \
  | grep -E "MEMORY|Failed|Exception|tool|Traceback"
```

Three things worth reporting back:

1. Any `[MEMORY] Failed to load history` or `[MEMORY] Failed to save
   message` lines — would confirm hypothesis 2.
2. Any tool-call exceptions — `MCPClient` errors, `cube_query_tool`
   timeouts, Aurora connection failures — confirms hypothesis 1.
3. The actual user message AgentCore received, vs what got logged in
   the chat UI. If they differ, the prompt-aggregator is dropping or
   transforming text — possibly the same UTF-8 bug from issue 1
   manifesting on the input side.

If hypothesis 2 is the cause, the cheapest workaround is to clear the
memory store and restart fresh:

```bash
MEMORY_ID=$(aws cloudformation describe-stacks \
  --stack-name agentic-analytics-demo --region us-west-2 \
  --query "Stacks[0].Outputs[?OutputKey=='MemoryId'].OutputValue" \
  --output text)

# List events for the test session and delete pre-fix ones
aws bedrock-agentcore memory list-events \
  --memory-id "$MEMORY_ID" --actor-id default --session-id default \
  --region us-west-2
# Then memory_delete_event by id, OR re-create the memory resource.
```

## Issue 3 — Guardrail false-positive on a clearly in-scope query

### What we saw

User: "Can you give me booking summary?"
Bot: "I can only help with unicorn rental analytics questions. Please
ask about bookings, revenue, customers, or unicorn management."

This is the **input redaction message** from `unicorn_rental_agent.py`
(`guardrail_redact_input_message` parameter). The user's question was
exactly within scope, so the guardrail is too aggressive.

### Where to look

`resources/agentic-analytics-workshop/infrastructure/stacks/agentcore-stack.yaml`

Search for `BedrockGuardrail` or `AWS::Bedrock::Guardrail`. The deny
topics, content filters, or word filters are probably catching the
word "booking" or "summary" via overlap with another topic
(financial-advice, profanity, etc.) or via a too-broad denyTopic
example.

### How to triage

```bash
GUARDRAIL_ID=$(aws cloudformation describe-stacks \
  --stack-name agentic-analytics-demo --region us-west-2 \
  --query "Stacks[0].Outputs[?OutputKey=='GuardrailId'].OutputValue" \
  --output text)

aws bedrock get-guardrail --guardrail-identifier "$GUARDRAIL_ID" \
  --region us-west-2
```

Look for:
- `topicPolicy.topics[*].definition` — too-broad definitions catch
  benign queries
- `contentPolicy.filters[*].inputStrength` — set to HIGH on
  HATE/INSULTS/etc. is fine; HIGH on MISCONDUCT often false-positives
- `wordPolicy` — managed word lists or any custom blocked words

Quickest workaround for the demo: lower `inputStrength` from `HIGH` to
`MEDIUM` on the offending filter, or remove the topic causing the
false-positive. Update the guardrail in place — no stack redeploy
needed.
